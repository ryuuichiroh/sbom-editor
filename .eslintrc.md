# ESLint & Prettier Configuration

This project uses ESLint and Prettier for code quality and formatting.

## Configuration Files

- `eslint.config.js` - ESLint configuration (flat config format)
- `.prettierrc.json` - Prettier formatting rules
- `.prettierignore` - Files to exclude from Prettier formatting

## Features

### ESLint
- **Strict TypeScript rules** enabled via `strictTypeChecked` and `stylisticTypeChecked`
- **React Hooks** validation
- **React Refresh** support for Vite
- **Prettier integration** to avoid conflicts
- **Type-aware linting** with project service

### Key Rules
- `@typescript-eslint/no-explicit-any`: Error - Prevents use of `any` type
- `@typescript-eslint/no-unsafe-*`: Error/Warn - Prevents unsafe type operations
- `@typescript-eslint/no-non-null-assertion`: Error - Prevents use of `!` assertion
- `@typescript-eslint/prefer-nullish-coalescing`: Error - Prefer `??` over `||` to avoid bugs with falsy values
- `@typescript-eslint/prefer-optional-chain`: Warn - Prefer `?.` over manual checks
- `@typescript-eslint/strict-boolean-expressions`: Off - Allows JavaScript's idiomatic truthy/falsy checks
- `@typescript-eslint/no-unnecessary-condition`: Off - Disabled due to false positives
- `react-hooks/exhaustive-deps`: Error - Validates hook dependencies
- `no-console`: Warn - Allows only `console.warn` and `console.error`

### Design Philosophy
- **Practical over pedantic**: Rules are enabled only when they provide clear value without excessive false positives
- **Idiomatic JavaScript/TypeScript**: Allows common patterns like `if (value)` for truthy checks
- **Type safety where it matters**: Strict rules for `any`, unsafe operations, and nullish coalescing
- **Developer experience**: Avoids rules that make code unnecessarily verbose

### Prettier
- Single quotes for strings
- Semicolons required
- 2-space indentation
- 100 character line width
- Trailing commas (ES5)
- LF line endings

## Available Scripts

```bash
# Run ESLint
npm run lint

# Run ESLint with auto-fix
npm run lint:fix

# Format code with Prettier
npm run format

# Check formatting without modifying files
npm run format:check
```

## IDE Integration

### VS Code
Install these extensions:
- ESLint (`dbaeumer.vscode-eslint`)
- Prettier (`esbenp.prettier-vscode`)

Add to `.vscode/settings.json`:
```json
{
  "editor.defaultFormatter": "esbenp.prettier-vscode",
  "editor.formatOnSave": true,
  "editor.codeActionsOnSave": {
    "source.fixAll.eslint": true
  },
  "eslint.validate": ["javascript", "typescript", "javascriptreact", "typescriptreact"]
}
```

## TypeScript Strict Mode

TypeScript strict mode is enabled in `tsconfig.app.json`:
- `strict: true` - Enables all strict type checking options
- `noUnusedLocals: true` - Reports unused local variables
- `noUnusedParameters: true` - Reports unused parameters
- `noFallthroughCasesInSwitch: true` - Reports fallthrough cases in switch

## Coding Conventions Enforced

Based on `10-coding-conventions.md`:
- ✅ TypeScript strict mode enabled
- ✅ Minimized use of `any` type (error level)
- ✅ Explicit type definitions encouraged (but not forced - type inference is allowed)
- ✅ Function components enforced (React Hooks rules)
- ✅ Named exports preferred (no specific rule, but convention)
- ✅ JSDoc for public APIs (encouraged but not enforced)
- ✅ Nullish coalescing (`??`) preferred over logical OR (`||`) to prevent bugs

## Why Some Rules Are Disabled

### `@typescript-eslint/strict-boolean-expressions` (Off)
**Reason**: JavaScript's truthy/falsy checks are idiomatic and widely understood.

**Example**:
```typescript
// Allowed (concise and clear)
if (value) { ... }

// Would be required if enabled (verbose)
if (value !== null && value !== undefined && value !== '') { ... }
```

**Trade-off**: We accept the risk of implicit type coercion for better code readability. The `prefer-nullish-coalescing` rule already prevents the most common bugs.

### `@typescript-eslint/no-unnecessary-condition` (Off)
**Reason**: High false positive rate, especially with:
- Union types
- Optional chaining
- Nullish coalescing operator

**Example of false positive**:
```typescript
const value = obj?.prop ?? 'default';  // ESLint incorrectly flags this
```

### `@typescript-eslint/explicit-function-return-type` (Off)
**Reason**: TypeScript's type inference is excellent. Explicit return types add noise without significant benefit in most cases.

**When to use explicit types**: Public APIs, exported functions, and complex return types.

## Troubleshooting

### ESLint errors about missing types
Ensure `tsconfig.json` is properly configured and all dependencies are installed.

### Prettier conflicts with ESLint
The configuration uses `eslint-config-prettier` to disable conflicting rules. If you see conflicts, ensure Prettier runs after ESLint.

### Performance issues
ESLint with type-checking can be slow on large projects. Consider:
- Using `eslint.lintTask.enable: true` in VS Code
- Running lint only on changed files in CI/CD
